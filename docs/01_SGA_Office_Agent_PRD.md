## 2. 产品概述 (Product Overview)

### 2.1 产品背景
随着大语言模型 (LLM) 与智能体 (Agent) 技术的飞速发展，AI 在企业协同、数据分析、公文处理等场景中的参与度日益加深。然而，大模型天然不擅长处理复杂的文档排版、公式计算以及非标准化的文件格式（如 Docx, Xlsx, PDF）。
SGA-Office 项目旨在打造一个完全 **“Agent-First（智能体优先）”** 的后端微服务平台。它充当了大模型的“机械臂”和“视觉神经”，让 Agent 可以无缝、精准地生成、解析、修改各类常见办公电子文档。

### 2.2 核心价值主张与设计原则
*   **弃用 UI，拥抱 API Contract**：彻底取消传统面向人类的 Web 交互逻辑，转而提供符合 **MCP (Model Context Protocol)** 规范的原子化工具（Tools）。
*   **场景化指令替代繁琐配置**：对大模型“降智减负”。Agent 仅需下达具体场景指令（如“在第三页插入此图片”或“使用该数据填充满这份报销单模板”），由底层引擎负责复杂的排版与坐标计算。
*   **支持多模态生态**：全面整合外部开源能力（如 Mermaid 流图、Draw.io 白板引擎、ECharts 等），将其转化为可以随时嵌入到最终报告中的富媒体资产。

## 3. 功能需求详细说明 (Functional Requirements)

将系统能力解耦为四大核心模块：Word 引擎、Excel 引擎、图形图表引擎、PDF 引擎。
*定义：【输入】要求 Agent 提供的参数形态；【输出】要求系统返回的数据或文件链接形态。*

### 3.1 模块一：智能文档组装与检索引擎 (Word / Docx)

此模块致力于将 Agent 生成的非结构化大纲或零散片段，组装为符合人类阅读审美的正式文件。

| 功能 ID | 功能名称 | 功能描述与 Agent 场景用例 | 输入 / 输出规范 | Priority |
| :--- | :--- | :--- | :--- | :--- |
| **DOC-01** | **结构化文本极速排版** | **描述**：提供将通用标记语言渲染为标准版式 Word 的能力。<br>**用例**：Agent 撰写完毕一份项目汇报 Markdown 大纲，请求系统转化为带标准标题层级和自动编号的正式 `.docx` 报告。 | **输入**: Markdown 文本。<br>**输出**: 新生成的 Docx 文件云端 URL。 | P0 |
| **DOC-02** | **占位符模板动态注水** | **描述**：基于已有定型的 Word 模板（如合同、红头文件），进行变量数据的精确替换。<br>**用例**：HR Agent 根据员工档案，自动替换《录用通知书.docx》中的 `{{姓名}}`、`{{部门}}` 字段，批量生成最终通知书。 | **输入**: 模板文件 URL, Key-Value JSON 字典。<br>**输出**: 替换后的 Docx 文件 URL。 | P0 |
| **DOC-03** | **富媒体元素动态落位** | **描述**：允许在已存在的 Docx 文件特定位置追加图片或外部生成的图表。<br>**用例**：数据 Agent 刚刚通过绘图引擎生成了一张营收柱状图，指令系统将其插入到业务分析报告的“第三章：财务表现”标题下方。 | **输入**: 源文件 URL, 目标段落关键词, 图片 URL。<br>**输出**: 修改后的 Docx 文件 URL。 | P1 |
| **DOC-04** | **长文本定向切片读取** | **描述**：规避超长上下文溢出，提供长文本特定章节的精准解析。<br>**用例**：法务 Agent 不需要通读 200 页合同材料，它向系统请求：“仅提取第四章违约责任的全部文本与表格”。 | **输入**: 源文件 URL, 章节名称/关键字检索。<br>**输出**: 纯文本或精简的 JSON/Markdown。 | P1 |

### 3.2 模块二：动态报表与数据处理器 (Excel)

此模块专注于结构化数据（表格）的高效存储、处理和二次利用。

| 功能 ID | 功能名称 | 功能描述与 Agent 场景用例 | 输入 / 输出规范 | Priority |
| :--- | :--- | :--- | :--- | :--- |
| **EXC-01** | **简单二维归档入库** | **描述**：将 Agent 收集到的一维/二维数据迅速固化为表格用于持久化留存。<br>**用例**：爬虫 Agent 将每日抓取的几十条新闻标题与链接，直接封存入一个标准的 Excel 记录表中。 | **输入**: 二维数组型 JSON。<br>**输出**: 基础 Excel 文件 URL。 | P0 |
| **EXC-02** | **存量表格增量追加** | **描述**：避免反复全量读写大型文件，支持针对特定 Sheet 表底追加新行。<br>**用例**：系统监控 Agent 每小时自动将异常日志汇总，并追加到本月的《系统巡检日志.xlsx》的末尾，不覆盖原数据。 | **输入**: 目标 Excel URL, 新增行 JSON 数组。<br>**输出**: 更新操作成功状态 (True/False)。 | P0 |
| **EXC-03** | **多维报表与公式生成** | **描述**：支持生成包含多个独立 Sheet、不同表头样式合并、以及预设计算宏/公式的行业级报表。<br>**用例**：财务 Agent 生成《月度结算单》，其中不仅包含原始数据，还包含了让财务人员可以二次核对的 `=SUM()` 动态汇总公式。 | **输入**: 多层嵌套的表结构的特定 JSON。<br>**输出**: 排版精美的 Excel 文件 URL。 | P1 |
| **EXC-04** | **命名区域精准解析** | **描述**：允许 Agent 选择性读取庞大算表中的有效数据区。<br>**用例**：Agent 需要根据去年的利润测算今年的人员编制，它请求：“仅读取 Sheet2 中包含‘一季度利润’的区域单元格数据”。 | **输入**: 目标 Excel URL, 目标 Sheet/命名范围。<br>**输出**: 结构化扁平 JSON。 | P2 |

### 3.3 模块三：业务图表与可视化引擎 (Diagram & Flowchart)

此模块是提升系统信息密度的核心。通过整合开源生态（如 Draw.io / Mermaid），弥补大模型视觉输出的短板。

| 功能 ID | 功能名称 | 功能描述与 Agent 场景用例 | 输入 / 输出规范 | Priority |
| :--- | :--- | :--- | :--- | :--- |
| **VIS-01** | **轻量流图即时直出** | **描述**：支持将 Mermaid 或 PlantUML 的类代码纯文本描述，静默渲染为可视化图形。<br>**用例**：技术支持 Agent 在回答某业务逻辑流转顺序时，自动输出了一段 Mermaid 代码，微服务秒级渲染出一张精美的时序图图片。 | **输入**: 特定标准语法的纯文本串。<br>**输出**: 高清 PNG/SVG 图片 URL。 | P0 |
| **VIS-02** | **商业统计图渲染计算** | **描述**：后台封装数据可视化图表库（如 ECharts/Matplotlib），根据统计数据自动绘图。<br>**用例**：销售 Agent 例行汇报某产品各区域销量，除了出具 Markdown 文字，同时配上系统刚刚由销量 JSON 绘制出的区域销量分布饼图。 | **输入**: 数据源 JSON, 图表类型 (如 bar, pie)。<br>**输出**: 图表截图 PNG/SVG URL。 | P1 |
| **VIS-03** | **可编辑互动白板生成** | **描述**：对接 Draw.io 等复杂开源组件协议，使大模型能够生成人类可二次编辑的高维图纸工程文件。<br>**用例**：产品 Agent 梳理了一份极度庞杂的业务状态机模型，请求导出为 Draw.io (XML格式) 源文件，以便真实的产品经理在 Web 端接手修改拓扑图的细节。 | **输入**: 符合架构解析器的节点关系 JSON。<br>**输出**: 原生工程文件格式 (如 `.drawio`)。 | P2 |

### 3.4 模块四：PDF 终态与合规组件 (PDF / Compliance)

此模块定义了最终文件的定稿、安全分发与溯源保障能力。

| 功能 ID | 功能名称 | 功能描述与 Agent 场景用例 | 输入 / 输出规范 | Priority |
| :--- | :--- | :--- | :--- | :--- |
| **PDF-01** | **多源文件定版转码** | **描述**：将易变文件转换为排版固定的 PDF 形态用于发送与归档。<br>**用例**：在多轮辩论结束后，双边 Agent 对某备忘录 (Docx 格式) 均无异议，调用接口定稿输出 PDF 分发给所有人类参会者。 | **输入**: Docx / Markdown / HTML 原文链接。<br>**输出**: 标准 PDF 文件 URL。 | P0 |
| **PDF-02** | **文件组密层防伪加印** | **描述**：不破坏文档正文流的前提下，完成底层加密标记动作。<br>**用例**：合规 Agent 在发出所有内部评审资料前，必须打上请求人姓名的全覆盖倾斜角半透明文字水印；或在收据指定发票专用章落位贴上透明图章。 | **输入**: 源 PDF URL, 水印文本 / 图章图片及坐标。<br>**输出**: 处理后的防伪 PDF URL。 | P1 |
| **PDF-03** | **物理拓扑层拼拆重组** | **描述**：精准控制大文件体积与内容外显。<br>**用例**：Agent 将两份各自独立的周度周报 PDF 拼接为一个双周报；或将长达 200 页的招股书仅裁取前 10 页执行摘要供初筛。 | **输入**: 源 PDF 列表, 页码区间指令。<br>**输出**: 重构后的单一 PDF URL。 | P1 |
| **PDF-04** | **复杂非结构化扫描解析** | **描述**：赋予 Agent 视觉“阅读”老旧/扫描文档的能力，基于 OCR+版面识别。<br>**用例**：审计 Agent 面对客户发来的一堆 2018 年往年模糊资产负债表扫描件（甚至有歪斜），调用引擎识别出文字和内部线条，将报表还原为可以用于逻辑运算的 JSON。 | **输入**: 图片格式 PDF 文件 URL。<br>**输出**: 版面结构化后的 Markdown 或表格 JSON。 | P2 |

## 4. 系统交互与非功能需求 (Non-Functional Requirements)

1. **环境与网络隔离 (COS/S3 Based I/O)**
   为了确保微服务可随意伸缩部署，以及 Agent 的运行时环境解耦，所有的文件交互严格禁止采用本地绝对路径。输入介质统一下载自公网可达 URL 或 Base64 流；所有持久化产物必须由系统自动上传至对象存储，并统一返回下载链接 (Access URL) 给调用方。

2. **异步状态机法则 (Async Workflows)**
   鉴于文件排版图表渲染具有极高的 CPU/内存开销。平台的所有涉及生成的接口（如 `VIS-X`、`DOC-X`）禁止阻断式同步等待。微服务在确认参数无误后立刻返回状态码 `HTTP 202 Accepted` 以及唯一任务跟踪标识 `TaskID`。需提供独立的 `/tasks/{task_id}/status` 轮询端口供 MCP Agent 检查进度。

3. **严格的契约约束 (Strict Typed Payload)**
   API 层需提供基于标准的 OpenAPI (Swagger) 接口定义文件。所有 JSON 参数（特别是涉及到表格结构 `schema_mapping` 这类易错字段）必须具备严格的边界条件检查规则和预编译示例，通过格式强制校验尽早阻断大模型因幻觉产生的残缺指令，系统直接丢弃脏数据并反馈错误定位。
